# 静态批调度与多客户端联邦 BO 帮助文档

## 1. 本次修改内容

本次在 `new_tr_split` 中新增了一条独立于动态窗口流式仿真的静态批调度实验路径：

| 文件 | 修改内容 |
| --- | --- |
| `static_batch_federated.py` | 新增静态批任务生成、批调度事件仿真、统一评价函数、多客户端 Local BO / Centralized BO / Federated BO。 |
| `runtime.py` | 把 `static_batch_federated.py` 加入运行时加载链。 |
| `cli.py` | 新增 `--mode batch_federated` 及一组静态批/多客户端/联邦 BO 参数。 |

旧的 `ConnectedFactory.run_continuous()` 动态窗口逻辑保留，不作为本入口的主流程。

## 2. 新流程

新流程从“动态窗口流式调度，任务可遗留到下一窗口”改为：

1. 生成若干异构客户端。
2. 每个客户端有自己的节点数量、边/云比例、网络延迟、带宽、算力、功耗、任务比例。
3. 每个客户端在一个批次开始时收到一批任务，任务 `create_time=0`。
4. 本地调度器按批内顺序逐个分派任务，执行完整事件仿真，直到该批任务完成。
5. 对该批任务计算统一 `cost`。
6. BO/CBO 用同一个 `cost` 做训练反馈和最终评价，避免训练/评估口径偏差。
7. 多客户端场景下，可以选择独立本地 BO、集中式 BO 上界、或联邦 BO。

## 3. 方法入口

统一命令入口：

```powershell
cd D:\CBOv2\新的代码结构
python -m new_tr_split --mode batch_federated --batch-method independent_bo
```

`--batch-method` 可选：

| 方法 | 含义 | 对应阶段 |
| --- | --- | --- |
| `fixed` | 固定默认权重，不做 BO。 | 阶段 1：单客户端/多客户端静态批调度基线 |
| `local_bo` | 只跑第一个客户端，本地 BO。 | 阶段 2：单客户端静态批 Local BO/CBO |
| `independent_bo` | 多客户端各自独立 Local BO，不共享信息。 | 阶段 3：多客户端独立 Local BO |
| `centralized_bo` | 所有客户端的 `(context, theta, cost)` 进入同一个中心 BO。 | 阶段 4：Centralized BO 上界 |
| `federated_bo` | 客户端保留本地 BO，只共享候选权重上的 surrogate 预测或经验摘要。 | 阶段 5：Federated BO / Fed-CBO |

说明：当 `--batch-context-features none` 时是普通 BO；选择情景信息后，就是静态批场景下的 CBO。

## 4. 常用示例

单客户端静态批固定权重：

```powershell
python -m new_tr_split --mode batch_federated --batch-method fixed --batch-clients 1 --batch-rounds 1 --batch-tasks 100
```

单客户端 Local BO，固定同一批任务，便于看 BO 是否能在静态批上收敛：

```powershell
python -m new_tr_split --mode batch_federated --batch-method local_bo --batch-clients 1 --batch-rounds 50 --batch-tasks 120 --batch-reuse-mode fixed_per_client
```

多客户端独立 BO：

```powershell
python -m new_tr_split --mode batch_federated --batch-method independent_bo --batch-clients 5 --batch-rounds 50 --batch-tasks 120
```

集中式 BO 上界：

```powershell
python -m new_tr_split --mode batch_federated --batch-method centralized_bo --batch-clients 5 --batch-rounds 50 --batch-tasks 120
```

联邦 surrogate BO：

```powershell
python -m new_tr_split --mode batch_federated --batch-method federated_bo --batch-clients 5 --batch-rounds 50 --batch-tasks 120 --batch-fed-share-mode surrogate
```

联邦经验摘要共享：

```powershell
python -m new_tr_split --mode batch_federated --batch-method federated_bo --batch-fed-share-mode experience
```

联邦 surrogate + 经验摘要混合：

```powershell
python -m new_tr_split --mode batch_federated --batch-method federated_bo --batch-fed-share-mode hybrid
```

## 5. BO 调节参数

BO 调节的仍然是通用调度器控制向量，不依赖客户端节点数量：

```text
W_RT_Latency
W_Batch_Latency
W_AI_Latency
W_RT_Energy
W_Batch_Energy
W_AI_Energy
W_Queue
W_Risk_Scale
Beta_Control
Opportunity_Rho
Cloud_Gate
```

这些参数对任意节点数量的客户端都通用。客户端异构性由节点配置、拓扑矩阵、任务比例、任务规模、网络/带宽倍率和情景信息表示。

## 6. 评价函数与归一化

静态批评价统一使用 `cost`，BO 训练反馈和最终 `batch_summary.csv` 评价使用同一个值。

默认：

```text
cost =
    delay_weight     * delay_norm
  + energy_weight    * energy_norm
  + lateness_weight  * lateness_norm
  + violation_weight * effective_vio_rate
  + makespan_weight  * makespan_norm
  + unfinished_weight * unfinished_rate
```

默认权重：

```text
delay=1
energy=0.25
lateness=2
violation=6
makespan=0.5
unfinished=20
```

可用参数覆盖：

```powershell
--batch-objective-weights delay=1,energy=0.2,lateness=2,violation=8,makespan=0.5,unfinished=20
```

归一化模式：

| 参数 | 含义 |
| --- | --- |
| `--batch-normalization batch_reference` | 默认。每个客户端/批次用任务和节点本身计算 reference，不使用任何方法运行结果。 |
| `--batch-normalization none` | 不归一化，直接使用原始量级。只建议做调试。 |

当前 `batch_reference` 的 reference 包括：

| reference | 来源 |
| --- | --- |
| `reference_delay` | 任务 deadline budget 与空载节点估计延迟的组合 |
| `reference_energy` | 空载节点对每个任务估计能耗的中位数均值 |
| `reference_lateness` | 平均 deadline budget |
| `reference_makespan` | deadline 分布和最短服务时间的批次参考 |

注意：这是一版保守归一化，重点是避免用某个方法的结果当归一化基准造成评估偏差。后续如果论文实验需要更严格的跨客户端可比性，可以把 reference 改成离线固定基准表或基于物理下界的校准表。

| 写法 | 含义 |
| --- | --- |
| `none` | 不使用情景信息，退化为普通 BO |
| `all` | 使用当前统计出的全部情景信息 |

## 8. 客户端和拓扑配置

快速生成异构客户端：

```powershell
--batch-clients 5
--batch-node-counts 6,8,10,12,7
--batch-topology-profile heterogeneous
```

`--batch-topology-profile` 可选：

| 值 | 含义 |
| --- | --- |
| `heterogeneous` | 默认混合边/云节点 |
| `edge_small` | 小边缘客户端，少量云节点 |
| `edge_large` | 边缘节点更多 |
| `cloud_heavy` | 云节点比例更高 |

不同客户端任务批大小：

```powershell
--batch-client-task-counts 80,120,160
```

不同客户端任务比例：

```powershell
--batch-client-task-probs 20,40,40;50,20,30;10,30,60
```

也可以用 JSON 完全指定客户端：

```json
{
  "clients": [
    {
      "name": "client_A",
      "node_ids": [0, 1, 4, 10],
      "task_count": 120,
      "task_probs": {"RT": 0.2, "Batch": 0.4, "AI": 0.4},
      "network_delay_scale": 1.0,
      "bandwidth_scale": 1.0,
      "task_scale": 1.0,
      "workload_scale": 1.0,
      "deadline_scale": 1.0
    }
  ]
}
```

运行：

```powershell
--batch-client-config path\to\clients.json
```

`node_ids` 使用默认 `CFG.NODES_CFG` 中的节点模板。也可以在 JSON 中直接给 `nodes` / `node_config`。

## 9. 批任务控制参数

| 参数 | 默认 | 含义 |
| --- | --- | --- |
| `--batch-reuse-mode` | `new_each_round` | 每轮新批次；`fixed_per_client` 表示每个客户端复用同一批任务。 |
| `--batch-order` | `deadline` | 批内本地调度顺序，可选 `generated/deadline/type_deadline/largest_workload/random`。 |
| `--batch-dispatch-gap` | `0.01` | 批内本地分派间隔。任务 `create_time` 仍为 0，用于让调度器看到局部队列变化。 |
| `--batch-deterministic-scheduler` | `true` | 默认关闭 Boltzmann 随机采样，降低 BO 反馈噪声。 |

## 10. 联邦配置

`--batch-fed-share-mode`：

| 值 | 含义 |
| --- | --- |
| `surrogate` | 客户端只上传候选 `theta` 上的 `mu/sigma` 预测，服务器聚合为 `mu_fed + beta * sigma_fed`。 |
| `experience` | 客户端共享少量 `(context, theta, cost)` 经验摘要，不共享原始任务/节点。 |
| `hybrid` | 同时使用 surrogate 和经验摘要。 |

其他参数：

| 参数 | 默认 | 含义 |
| --- | --- | --- |
| `--batch-fed-candidates` | `96` | 每轮联邦 surrogate 聚合的候选权重数量。 |
| `--batch-fed-beta` | `CFG.FED_BETA` | 联邦 UCB 的探索系数。 |

## 11. 输出文件

默认输出目录：

```text
results/static_batch_<method>_seed<seed>
```

如果传 `--output-root`，则写入指定目录。

主要文件：

| 文件 | 内容 |
| --- | --- |
| `batch_detail.csv` | 每个 round/client 的详细指标、cost、theta、context、fed 诊断。 |
| `batch_summary.csv` | 每个客户端和 ALL 汇总的 mean/best/final cost、delay、energy、violation、completion。 |
| `client_configs.csv` | 本次生成或读取的客户端配置摘要。 |
| `run_config.json` | 本次运行参数、可选情景信息列表、归一化说明。 |

## 13. 后续建议

1. 先用 `fixed_per_client` 验证单客户端静态批 BO 是否稳定优于固定权重。
2. 再用 `new_each_round` 验证跨批次泛化。
3. 对比 `independent_bo / centralized_bo / federated_bo` 的早期轮次 cost，重点看新客户端冷启动收益。
4. 若论文需要严格跨客户端归一化，建议把 `batch_reference` 替换或增强为固定校准表，并记录校准数据来源。
