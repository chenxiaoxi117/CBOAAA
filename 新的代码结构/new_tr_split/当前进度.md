## 当前结果口径

结果路径中出现：

```text
task_effective_backlog_violation
```

说明这批实验的 BO/CBO 训练反馈采用修改后的训练目标，而不是最早的 `window_original`。

需要区分两个 cost：

1. `Eval_Cost` / `mean_eval_cost`

这是按真实窗口仿真结果计算的评估 cost。也就是说，每一轮都真实运行一个 BO window，记录实际到达、完成、延迟、能耗、违规和积压后，再由 `WindowSnapshot.to_metrics()` 计算。

默认窗口评估 cost 为：

```text
Eval_Cost =
    avg_energy
  + ALPHA_LATENCY * avg_delay
  + SLA_PENALTY_WEIGHT * vio_rate
  + LATE_PENALTY_WEIGHT * avg_lateness
  + BACKLOG_WEIGHT * backlog
  + zero_completion_penalty
  - early_bonus
```

当前默认主要权重：

```text
ALPHA_LATENCY = 100
SLA_PENALTY_WEIGHT = 1500
LATE_PENALTY_WEIGHT = 300
BACKLOG_WEIGHT = 200
ZERO_COMPLETION_PENALTY = 5000
USE_EARLY_BONUS = False
```

因此表格里的：

```text
mean_eval_cost
mean_delay
mean_backlog
mean_violation
mean_energy
```

都是基于真实窗口运行后的评估指标。

2. `BO_Training_Cost`

这是 BO/CBO agent 用于 `tell()` 更新 GP 的训练反馈。当前路径显示使用：

```text
task_effective_backlog_violation
```

对应训练 cost 为：

```text
BO_Training_Cost =
    avg_energy
  + ALPHA_LATENCY * avg_delay
  + LATE_PENALTY_WEIGHT * avg_lateness
  + REF_SIMPLE_BACKLOG_WEIGHT * BACKLOG_WEIGHT * backlog
  + SLA_PENALTY_WEIGHT * violation
```

其中 backlog 是轻量加入，默认 `REF_SIMPLE_BACKLOG_WEIGHT = 0.25`。

## 当前判断

这批 7 方法实验是有代码对应的，且 CBO 方法确认为：

```text
reduced6_cbo_lite_pressure_taskmix_counts
```

从结果路径中的 `task_effective_backlog_violation` 可以判断：这批结果使用的是修改后的 BO 训练反馈口径。

但表中的 `mean_eval_cost` 仍然是基于真实窗口仿真的评估 cost，不是直接拿训练反馈 cost 做最终评价。

1. 任务模型从旧的 `data / cpu / dur / deadline_factor` 扩展为：
   `upload_size / download_size / workload_cycles / required_cores / deadline / delay_sensitivity / task_type`。
   旧字段仍保留为兼容别名，避免已有脚本和诊断入口直接失效。

2. 节点模型从旧的 `cpu / speed / p_idle / p_max` 扩展为：
   `num_cores / capacity_slots / service_rate_gips / idle_power / max_power / uplink_bandwidth / downlink_bandwidth / propagation_delay / accelerator_type / reserved_resource`。
   其中 `capacity_slots` 表示抽象并发资源槽位或资源池能力，不再解释为真实物理核数；`service_rate_gips` 表示 Gcycles/s 级服务率，不再解释为 CPU GHz。

3. 任务总时延改为显式四段：
   `T = T_upload + T_queue + T_compute + T_download`。
   事件流也同步改成：任务到达 -> 上传完成 -> 进入计算队列 -> 计算完成 -> 下载完成 -> 任务完成统计。

4. 能耗拆成计算功耗积分和传输能耗：
   计算能耗仍按节点功率-时间积分统计，传输能耗分别按上传和下载数据量累计。

5. RT / Batch / AI 的异构匹配机制改为更容易解释的物理机制：
   RT 主要通过低网络时延、deadline 风险、短队列和 `reserved_resource` 资源预留体现；
   Batch 通过吞吐型节点和多槽位容量体现；
   AI 通过 `accelerator_type` 和可选 `accel_factor` 消融体现。
   默认仍关闭 `accel_factor`，避免主实验依赖人为倍率。

6. 压力扫描输出增加负载校准字段：
   `Expected_Tasks_Per_Window = lambda * BO_INTERVAL`、平均利用率、总完成比例、窗口完成比例、积压增长率等。
   这样后续解释 `lambda=1.8/2.6/3.0` 时，可以用系统实际利用率、完成率和积压增长来定义低/中/高或过载场景。

### 是否影响现在的结果

会影响“重新运行后的实验结果”，但不会修改已经保存下来的旧结果文件。

具体来说：

1. 已经跑出来的 CSV、图和历史结果不会自动改变。
2. 如果用修改后的代码重新跑同样命令，结果会改变，因为任务时延、传输能耗、节点容量解释和 RT 预留机制都已经变了。
3. 旧结果和新结果不应混在同一个主实验表里直接比较；如果论文主实验采用新建模层，baseline、BO、CBO、direct heuristic 都应该在新建模层下统一重跑。
4. CBO 上层逻辑没有被重写，但它接收到的窗口 cost、delay、energy、backlog、violation 等反馈会变，因此 BO/CBO 的搜索轨迹和最终策略也会随之改变。
5. 如果只是检查算法代码结构，影响不大；如果是复现实验数值或画主论文图，必须把本次修改视为一次新的实验设定版本。
