\section{动态云边端调度实验修改指南} \subsection{一、问题定义} 当前实验不建议再表述为纯静态批调度，也不建议仅用几个固定任务比例来证明动态性。建议将问题统一定义为： \begin{quote} 本文研究工业云边端环境下的动态序贯任务调度问题。每个调度窗口或 episode 内，任务按到达流或批内释放时间进入系统，底层调度器逐任务选择执行节点，并根据节点队列、资源占用、预计完成时间和 deadline 风险动态更新系统状态；上层 CBO 根据任务结构、负载压力和资源状态等上下文信息，自适应调整底层调度器的权重参数。 \end{quote} 需要避免两种不稳的表述： \begin{itemize} \item 不要说所有任务一次性静态分配； \item 不要只因为设置了 RT-heavy、AI-heavy、Batch-heavy 等固定比例，就声称方法验证了动态适应能力。 \end{itemize} 如果要证明动态性，必须在实验中加入随时间变化的因素，例如负载变化、任务比例变化、资源扰动、新任务突发到达或 deadline 压力变化。 \subsection{二、节点与任务建模修改} \subsubsection{1. 节点模型} 节点不应继续使用容易误解的 \texttt{speed/cpu} 表达。建议改为如下字段： \begin{verbatim} node_id node_type # edge / fog / cloud / ai_accel / rt_edge capacity_slots # 并发资源槽位或 VM slots service_rate_gips # Gcycles/s，不能解释为 GHz memory_gb idle_power max_power uplink_bandwidth downlink_bandwidth propagation_delay accelerator_type task_node_affinity_factor \end{verbatim} 其中，\texttt{service_rate_gips} 表示抽象服务率，不是 CPU 主频；\texttt{capacity_slots} 表示资源池或虚拟资源槽位，不能简单解释为物理核心数。 \subsubsection{2. 任务模型} 任务不应只保留 RT、Batch、AI 三个标签。每类任务至少应包含： \begin{verbatim} task_type # RT / Batch / AI workload_cycles upload_size download_size deadline memory_demand delay_sensitivity arrival_time \end{verbatim} 计算时间和传输时间建议写成： \[ T^{comp}_{i,j} = \frac{C_i}{\mu_j A_{j,k}} \] \[ T^{upload}_{i,j} = \frac{D^{up}_i}{B^{up}_j} + \tau_j \] \[ T^{download}_{i,j} = \frac{D^{down}_i}{B^{down}_j} + \tau_j \] 其中： \begin{itemize} \item $C_i$ 表示任务计算量； \item $\mu_j$ 表示节点基础服务率； \item $A_{j,k}$ 表示节点 $j$ 对任务类型 $k$ 的适配系数； \item $D^{up}_i,D^{down}_i$ 表示上传和下载数据量； \item $B^{up}_j,B^{down}_j$ 表示节点链路带宽； \item $\tau_j$ 表示传播时延。 \end{itemize} \subsubsection{3. 任务--节点适配矩阵} 如果正式实验启用 \texttt{task-adaptation} 或 \texttt{accel\_factor}，必须将其定义为环境异构性，而不是 CBO 独有机制。 建议命名为： \begin{verbatim} task_node_affinity_factor \end{verbatim} 或： \begin{verbatim} task_specific_service_factor \end{verbatim} 公式为： \[ \mu_{j,k}^{eff} = \mu_j \cdot A_{j,k} \] 其中 $A_{j,k}$ 表示节点 $j$ 执行任务类型 $k$ 时的有效服务率修正。 实验要求： \begin{itemize} \item 所有 baseline 和 CBO 必须共享同一套适配矩阵； \item 主实验命令必须显式记录 \texttt{task\_adaptation=True}； \item 需要补充 \texttt{with task adaptation} 与 \texttt{without task adaptation} 消融实验。 \end{itemize} \subsection{三、底层节点评分归一化} 底层调度器每次为一个任务选择节点时，应在当前任务的候选节点集合内归一化，而不是使用全局固定尺度。 对于任务 $i$ 和候选节点 $j$，先估计： \[ D_{i,j},\quad E_{i,j},\quad Q_j,\quad R_{i,j} \] 其中： \begin{itemize} \item $D_{i,j}$：任务 $i$ 放到节点 $j$ 的预计完成时延； \item $E_{i,j}$：预计能耗； \item $Q_j$：节点队列或负载压力； \item $R_{i,j}$：deadline 风险。 \end{itemize} 推荐归一化方式： \[ D^{ddl}_{i,j} = \frac{D_{i,j}}{deadline_i+\epsilon} \] \[ \tilde D_{i,j} = \frac{ D^{ddl}_{i,j} - \min_k D^{ddl}_{i,k} }{ \max_k D^{ddl}_{i,k} - \min_k D^{ddl}_{i,k} + \epsilon } \] \[ \tilde E_{i,j} = \frac{ E_{i,j} - \min_k E_{i,k} }{ \max_k E_{i,k} - \min_k E_{i,k} + \epsilon } \] \[ \tilde Q_j = \frac{used\_slots_j}{capacity_j+\epsilon} \] \[ \tilde R_{i,j} = \min\left( 1, \max\left( 0, \frac{D_{i,j}-deadline_i}{deadline_i+\epsilon} \right) \right) \] 节点得分函数改为： \[ S_{i,j} = w_D \tilde D_{i,j} + w_E \tilde E_{i,j} + w_Q \tilde Q_j + w_R \tilde R_{i,j} + w_C I^{cloud}_j \] 最终选择： \[ j^* = \arg\min_j S_{i,j} \] 注意：底层节点评分归一化只用于当前任务选节点；窗口级评价函数需要单独归一化，二者不能混为一谈。 \subsection{四、窗口级评价函数修改} 原始形式： \[ J = \alpha \bar E + \beta \bar D + \gamma V + \delta L + \eta B + P_0 \] 不建议继续直接使用，因为能耗、时延、违约率、超期时间和积压数量的量纲不同，且 $D,V,L,B$ 之间存在重复惩罚。 建议改成： \[ J_t = w_D \tilde D_t + w_E \tilde E_t + \lambda_V \max(0,V_{RT,t}-\epsilon_{RT}) + \lambda_U U_t + \lambda_B \max(0,\Delta B_t) \] 其中： \[ \tilde D_t = \frac{1}{N_t^{arr}} \sum_{i \in A_t} \frac{T_i^{cens}}{deadline_i+\epsilon} \] \[ \tilde E_t = \frac{E_t^{window}}{N_t^{arr}+\epsilon} \] \[ V_{RT,t} = \frac{N_{RT,t}^{violated}}{N_{RT,t}^{arr}+\epsilon} \] \[ U_t = \frac{N_t^{unfinished}}{N_t^{arr}+\epsilon} \] \[ \Delta B_t = \frac{B_t-B_{t-1}}{N_t^{arr}+\epsilon} \] 其中： \begin{itemize} \item $A_t$ 表示窗口 $t$ 内到达的任务集合； \item $N_t^{arr}$ 表示窗口内到达任务数； \item $T_i^{cens}$ 表示任务完成时延；若窗口结束时未完成，则用截尾时延； \item $V_{RT,t}$ 表示 RT 任务违约率； \item $U_t$ 表示当前窗口新到达任务未完成比例； \item $\Delta B_t$ 表示积压增长率。 \end{itemize} 关键要求： \begin{itemize} \item 不要只对 completed tasks 计算平均时延和能耗； \item 不要直接使用 raw backlog； \item lateness 可作为诊断指标单独报告，不建议与 violation、delay、backlog 重复重罚。 \end{itemize} \subsection{五、动态场景归一化方案} 动态场景下，不建议使用全局最大最小值归一化，也不建议用测试集未来信息。建议使用“场景校准尺度”。 流程如下： \begin{verbatim} 识别新场景 ↓ 使用预设 probe policy / fixed-mid / greedy-cost 运行前 K 个校准窗口 ↓ 统计 D_scale, E_scale, L_scale 等归一化尺度 ↓ 冻结这些 scale ↓ CBO、BO、fixed、greedy 等所有方法统一使用该 scale ↓ 最终评价阶段基于实际运行结果重新计算 raw metrics 和 normalized cost \end{verbatim} 例如： \[ D_s^{scale} = median(D_1,D_2,\dots,D_K) \] \[ E_s^{scale} = median(E_1,E_2,\dots,E_K) \] 后续窗口： \[ \hat D_t = \frac{\tilde D_t}{D_s^{scale}+\epsilon} \] \[ \hat E_t = \frac{\tilde E_t}{E_s^{scale}+\epsilon} \] 如果动态场景中途发生明显变化，例如： \begin{itemize} \item balanced $\rightarrow$ RT-heavy； \item low-load $\rightarrow$ high-load； \item AI 节点降速； \item 边云链路延迟升高； \end{itemize} 则应按阶段重新校准，或将实验明确划分为多个动态阶段。 注意： \begin{quote} 该尺度应称为“场景归一化参考尺度”，不能称为“CBO 的基线”。所有方法必须共享同一套冻结尺度。 \end{quote}
