# V1.1 修改说明

## 1. 问题定义与默认场景

本版本将主实验口径统一为“动态云边端序贯任务调度”：任务按到达流进入系统，底层调度器逐任务选择节点，上层 BO/CBO 根据窗口前可观测上下文调整权重。

默认 `LAMBDA_SCHEDULE` 已由单一静态到达强度改为三阶段动态负载：

```text
0-40000s: lambda=1.0
40000-80000s: lambda=1.8
80000-120000s: lambda=1.2
```

固定 RT/Batch/AI 比例仍可作为场景结构，但不再单独作为“动态性”证据。

## 2. 节点与任务模型

节点配置改为以 `node_type / capacity_slots / service_rate_gips / memory_gb / idle_power / max_power / uplink_bandwidth / downlink_bandwidth / propagation_delay / accelerator_type / task_node_affinity_factor` 为主字段。

任务配置补齐 `memory_demand` 和 `arrival_time`，主字段为 `task_type / workload_cycles / upload_size / download_size / deadline / memory_demand / delay_sensitivity / arrival_time`。

旧字段读取仍保留兼容，但主配置不再写 `cpu/speed` 作为规范字段。`task_node_affinity_factor` 默认启用，表示共享环境异构矩阵，不是 CBO 独有机制；可用 `--no-task-adaptation` 做消融。

## 3. 底层节点评分

默认 `SCHEDULER_SCORE_NORM_MODE` 改为 `candidate_minmax_deadline`。

每次任务选节点时，仅在当前候选节点集合内归一化：

```text
delay: response_time / deadline_budget 后做候选集 min-max
energy: 预计能耗做候选集 min-max
queue: used_slots / capacity_slots
risk: deadline overrun ratio 截断到 [0, 1]
cloud: 可选 cloud penalty
```

Constrained Boltzmann、普通 Boltzmann 以及 direct greedy / queue-aware baseline 均使用同一套候选归一化口径。

## 4. 窗口级评价函数

`WindowSnapshot.to_metrics()` 的主 `cost` 已改为按本窗口到达任务计算：

```text
cost =
  w_D * window_delay_deadline_norm
  + w_E * window_energy_norm
  + lambda_V * max(0, window_rt_violation_rate - eps_rt)
  + lambda_U * window_unfinished_rate
  + lambda_B * max(0, window_backlog_growth_rate)
```

未完成任务使用窗口结束时刻的截尾时延。`avg_lateness`、completed-only delay、raw backlog 等仍作为诊断指标导出，但不再进入主窗口 cost 的重复惩罚。

## 5. 场景归一化参考尺度

`CBO_REFERENCE_MODE` 默认改为 `calibrate`。场景实验开始前会用固定 probe 策略（优先 `reduced6_fixed_mid`）运行前 K 个校准窗口，生成并冻结同一套 `scenario_normalization_reference_scale`，随后 Fixed、BO、CBO、direct baselines 共享该参考尺度。

输出文件新增：

```text
scenario_normalization_reference.json
```

运行配置中也会显式记录：

```text
use_task_type_adaptation: true
task_adaptation_field: task_node_affinity_factor
scenario_normalization_reference: ...
```


## 7. V1.1 补充：训练目标与 phase reference

本次补充将主实验默认训练目标从 `eval_cost` 切换为：

```text
CBO_OBJECTIVE_MODE = "normalized_tradeoff"
```

BO/CBO 的方向已核对：外部 `train_cost` 仍然是越小越好；`agent.tell(theta, cost)` 内部记录 `y = -cost`；acquisition 最大化 `y`。因此 `normalized_tradeoff_score` 作为越小越好的 cost 输入时，优化方向一致。

动态 reference 不再按简单 low / medium / high 标签切换，而是按外部场景是否显著改变来生成 `phase_signature`：

```text
lambda_level / task_mix_level / deadline_pressure_level
resource_perturbation_id / link_profile_id / task_adaptation_enabled
```

其中 `resource_perturbation_id` 默认使用 `normal`，建议后续资源扰动命名采用 `ai_capacity_degraded` 或 `ai_accel_availability_low`，避免被误解为硬件频率下降。

`phase_signature` 只描述外部场景条件，不包含 backlog、violation、completion rate、当前 cost 等算法运行结果。backlog 继续保留在 context、window cost 和诊断指标中，但不作为 reference key。

默认显著变化阈值：

```text
lambda_relative_change >= 0.30
task_mix_l1_distance >= 0.25
deadline_pressure_relative_change >= 0.20
resource_perturbation_id / link_profile_id / task_adaptation_enabled 发生变化
```

动态实验中会生成：

```text
reference_cache[phase_signature] = frozen_reference
```

相近场景复用同一个 `active_reference_id`；显著不同场景才新建 reference。校准窗口由 probe policy 在正式实验前单独运行，不混入主结果；日志字段新增：

```text
phase_id
phase_signature
active_reference_id
reference_source
is_calibration_window
calibration_window_label
phase_reference_cache_status
```

输出文件新增或扩展：

```text
scenario_normalization_reference.json
scenario_normalization_reference_cache.json
refactor_run_config.json
dynamic_run_config.json
```

## 8. alpha 重参数化与 fixed-tuned 口径

`alpha_direct` 被明确为重参数化方案：

```text
w_rt_delay = alpha_rt,       w_rt_energy = 1 - alpha_rt
w_batch_delay = alpha_batch, w_batch_energy = 1 - alpha_batch
w_ai_delay = alpha_ai,       w_ai_energy = 1 - alpha_ai
```

默认 alpha 搜索范围调整为：

```text
alpha_rt:    0.70 - 0.98
alpha_batch: 0.50 - 0.90
alpha_ai:    0.50 - 0.95
```

如果仍使用 legacy 独立权重模式，则继续通过 latency / energy / queue / risk 的独立 bounds 做 sweep，并检查 top theta 边界命中率和 score 分项贡献。

由于训练目标、score 归一化、task adaptation 默认值和 alpha bounds 均已变化，`fixed-tuned` 必须用新目标和新 bounds 重新调参后再作为强基线使用。

## 9. reduced7 / reduced9 能耗权重扩展

本次新增两种不推翻现有调度器的能耗权重控制方案：

```text
reduced7:
[
  W_RT_Latency,
  W_Batch_Latency,
  W_AI_Latency,
  W_Queue,
  W_Risk_Scale,
  Cloud_Gate,
  W_Energy_Scale
]
```

`reduced7` 将 `W_Energy_Scale` 同时映射到 `W_RT_Energy / W_Batch_Energy / W_AI_Energy`，用于回答“固定能耗权重 1.5 是否不合理，是否应该让 BO/CBO 学一个全局节能倾向”。

```text
reduced9:
[
  W_RT_Latency,
  W_Batch_Latency,
  W_AI_Latency,
  W_RT_Energy,
  W_Batch_Energy,
  W_AI_Energy,
  W_Queue,
  W_Risk_Scale,
  Cloud_Gate
]
```

`reduced9` 支持不同任务类型学习不同能耗敏感性，适合作为增强消融。由于维度从 6 增加到 9，它暂不建议直接替代主线，应先观察 500 轮内的收敛稳定性、边界命中率和分项贡献。

新增可选方法：

```text
reduced7_fixed_mid
reduced7_fixed_tuned
reduced7_bo_greedy
reduced7_cbo_lite_pressure_taskmix_counts

reduced9_fixed_mid
reduced9_fixed_tuned
reduced9_bo_greedy
reduced9_cbo_lite_pressure_taskmix_counts
```

新增常用别名：

```text
reduced7-bo-greedy
reduced7-cbo / reduced7-cbo-ptc / cbo7
reduced9-bo-greedy
reduced9-cbo / reduced9-cbo-ptc / cbo9
```
